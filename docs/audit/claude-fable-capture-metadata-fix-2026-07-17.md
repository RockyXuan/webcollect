> 主线采纳说明（2026-07-23）：本文准确定位了浮窗 GitHub 标题、平台通用简介、缺少 README 抓取以及重复网址被静默跳过四个根因。本文原建议“重复网址直接更新”没有用户确认和并发保护，不符合最终产品约束；V1.5.2 改为浮窗新旧对比、用户明确确认，并在入库前复核卡片 ID、网址和 `updatedAt`。以 V1.5.2 closeout 与当前代码为最终准则。

## 一句话结论

用户遇到的标题、简介问题背后有四个独立根因，其中最关键的是：

> 收藏浮窗保存后，队列 drain 发现 URL 已存在便跳过、不更新现有卡片。用户在浮窗里改的标题和简介被静默丢弃，看起来像“改了没反应”。

## 用户诉求拆解

| 编号 | 类别 | 需求 | 现象 |
|---|---|---|---|
| 1a | 标题质量 | 收藏 GitHub 仓库时标题为项目名，例如 `codex-slides` | 标题是超长的 owner、repo 和宣传语 |
| 1b | 标题保存 | 手动改标题保存后应生效 | 去 App 看仍是旧标题 |
| 2a | 简介质量 | 读 README 凝练项目自身摘要 | 翻译得到 GitHub 平台通用介绍 |
| 2b | 简介保存 | 翻译或编辑后的简介应保存 | App 里的详细介绍仍为空 |

## 数据流与根因

```text
Chrome 扩展浮窗
  初始标题/简介：浮窗自身的 compactCaptureTitle 与页面描述
  异步抓取：background FETCH_META
  保存：最终草稿进入浮窗队列
    ↓
队列 drain
  URL 已存在：旧实现直接标成 imported 并跳过
    ↓
App 卡片
  现有卡片保持旧标题和旧简介
```

### 根因 1：浮窗标题缺少 GitHub 仓库规则

共享元数据提取器已能从 GitHub 仓库网址取仓库名，但浮窗自己的 `compactCaptureTitle` 没有相同规则。初始长标题又会使后续正确仓库名无法覆盖，因此浮窗仍显示很长的 GitHub 页面标题。

### 根因 2：重复网址在 drain 中被静默跳过

旧逻辑只要发现规范化 URL 已存在，就把队列项标成已导入并写入 `Duplicate URL skipped`，现有卡片完全不动。浮窗中用户修改的标题和简介因此无法落到卡片。

### 根因 3：具体仓库错误套用 GitHub 平台通用摘要

本地翻译模块对所有 `github.com` 页面优先返回“代码托管与协作平台”一类通用介绍。它对 GitHub 首页合理，对具体项目仓库不合理。

### 根因 4：没有读取 README 的能力

旧版只使用页面 OG/About 描述，从未读取公开 README，因此无法得到项目自身更有意义的首段介绍。

## 原审查建议

1. 修复重复 URL 处理，让浮窗改动可以更新现有卡片，同时保留卡片位置和用户字段。
2. GitHub 仓库及子页面标题直接取仓库名，排除用户主页和保留路径。
3. GitHub 平台级通用摘要只用于首页，不用于具体仓库。
4. 通过 `raw.githubusercontent.com` 读取公开 README，跳过徽章、图片、目录、代码和安装命令，提取第一段实质介绍；失败回退页面描述。
5. 不引入 GitHub Token、付费翻译服务或 AI。

## V1.5.2 安全修订

主线保留以上元数据方向，但把重复网址写入约束收紧为：

- 保存前只读查询现有卡片。
- 只有一张重复卡片时，在当前浮窗显示原内容与新内容。
- 用户选择“更新原卡片”后，草稿携带目标卡片 ID 和确认时的版本。
- drain 再次核对 ID、规范化 URL 和 `updatedAt`；冲突即停止，不覆盖。
- 只更新非空标题、`shortDesc` 和 `fullDesc`。
- 保留 ID、URL、分项、分类、顺序、备注、简称、创建时间和图标。
- 多个重复卡片不自动选择；旧队列记录与无浮窗右键收藏继续安全跳过。
- 内容实际变化时才触发业务保存、同步版本和 dirty 标记。

## 验证要求

- GitHub 仓库、子页面、用户主页和保留路径识别。
- README 正常、无 README、徽章密集、超时、超限和格式异常。
- GitHub 首页继续使用平台摘要，具体仓库不使用。
- 单重复确认更新、取消、版本冲突、多重复、空字段和相同内容。
- 更新后分类、顺序、备注、简称、图标与创建时间不变。
- 不新增权限、存储 key、Drive Schema、依赖或数据迁移。
- 不清空、覆盖或批量改写用户收藏、分类、分项、回收站、快照、同步状态和 seed。
